# 第7回ハンズオン：集中型VCSのメリットをシミュレーションする

## 概要

第7回「集中型VCSの設計哲学——それは本当に『悪』だったのか？」に対応するハンズオン演習。

集中型VCSの構造的なメリット——中央リポジトリによる一貫性の保証、パスベースアクセス制御、単一信頼点（Single Source of Truth）——を実際の操作で体験する。同時に、サーバ停止時の影響を確認し、集中型の設計トレードオフを手で確かめる。

## 学べること

- 中央リポジトリが唯一の信頼できる情報源として機能する仕組み
- コミットの即時反映による一貫性の体験
- Subversionのパスベースアクセス制御（authzファイル）の設定方法
- 中央サーバが利用不可な場合の影響と制約
- 集中型VCSと分散型VCSの設計選択の比較

## 動作環境

- Linux（Ubuntu 24.04 推奨）
- macOS（Homebrew で `brew install cvs subversion`）
- Windows（WSL2 推奨）

Docker を使う場合:

```bash
docker run -it --rm ubuntu:24.04 bash
apt update && apt install -y cvs subversion
```

## 実行方法

```bash
bash setup.sh
```

## 演習内容

| 演習 | テーマ                 | 概要                                                           |
| ---- | ---------------------- | -------------------------------------------------------------- |
| 1    | 中央リポジトリの唯一性 | 同一リポジトリから複数の作業コピーを作成し、中央の唯一性を確認 |
| 2    | コミットの即時反映     | 一方のコミットが他方のupdateで即座に反映されることを確認       |
| 3    | パスベースアクセス制御 | SVNのauthzファイルによるディレクトリ単位の権限設定を体験       |
| 4    | サーバ停止の影響       | 中央サーバが利用不可な場合のコミット・ログ参照の失敗を確認     |
| 5    | 設計選択の比較         | 集中型と分散型の設計トレードオフを表で整理                     |
