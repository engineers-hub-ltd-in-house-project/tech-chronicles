# Claude向け執筆ガイド

**Anthropic Claude（Opus / Sonnet）を使った技術史連載の執筆方法**

---

## Claudeの特性と強み

Claude（特にOpus）は、本リポジトリの執筆ワークフローに最も適したモデルの一つである。以下の特性が技術史連載の執筆と相性が良い。

- **長文指示への忠実性**: 指示書全文（数万字）を一度に渡しても、細部まで遵守する
- **文体の再現精度**: 文体サンプルを複数提示すると、そのトーン・リズムを高精度に再現する
- **構造化出力**: 「5部構成テンプレート」のような構造的要件を安定して満たす
- **Deep Research機能**: ファクトチェック工程をモデル内で完結できる
- **ファイル生成**: Markdown、PDF等のファイルを直接生成・ダウンロード可能

---

## 推奨ワークフロー

### Step 1: 指示書を一括投入する

Claudeは長いコンテキストウィンドウを持つ。指示書の全文を**最初のメッセージで一括投入**するのが最も効果的。

```
【実践手順】
1. blueprints/ 配下の指示書（例: version-control.md）全文をコピー
2. Claudeの新しい会話に貼り付ける
3. 指示書の後に「この指示書に基づいて執筆を行います」と添える
```

**ポイント**: Claudeは最初のメッセージの指示を会話全体を通じて記憶する。途中で追加指示するよりも、最初に全量を渡す方が一貫性が高い。

### Step 2: アウトラインを確認する

いきなり本文を書かせず、まずアウトラインを生成させる。

```
プロンプト例：
「第1回『なぜこの連載を書くのか——git ありきの世界への違和感』の
 アウトラインを、指示書のテンプレートに沿って作成してください。
 各セクションの想定文字数と、扱う具体的なトピックを含めてください。」
```

アウトラインを確認し、修正点があれば伝える。「良さそうです」と承認してから次に進む。

### Step 3: ファクトチェックを実行する

Claude（特にclaude.aiのDeep Research機能）は、Web検索を行いながら事実確認ができる。

```
プロンプト例：
「第1回の執筆に必要な以下の事実・統計データを正確に調査してください。
 各項目について、可能な限り一次ソース（公式ブログ、公式発表、論文等）に
 基づいた情報を提供してください。

 1. GitHubの最新の開発者数（公式発表値）
 2. Stack Overflow Developer Surveyにおけるgitの使用率
 3. Gitの誕生日（最初のコミット日）の正確な日時
 ...」
```

**ポイント**: 調査項目を番号付きリストで明示する。「全般的に調べて」ではなく、具体的に何を知りたいかを列挙する。

### Step 4: 本文を執筆させる

アウトライン承認＋ファクトチェック完了後、本文を依頼する。

```
プロンプト例：
「一記事目を書いてください」
```

Claudeは指示書・アウトライン・ファクトチェック結果を会話コンテキスト内に保持しているため、シンプルな指示で十分。

### Step 5: Markdownファイルとして出力する

Claudeはファイルを直接生成できる。

```
プロンプト例：
「この記事をMarkdownファイルとしてダウンロードできるようにしてください」
```

## Claude固有のテクニック

### テクニック1: 「禁止事項」を明示する

Claudeは「やるべきこと」と同様に「やってはいけないこと」にも忠実に従う。指示書の「やらないこと」リストが有効に機能する。

```markdown
#### やらないこと：

- git/GitHubの礼賛記事にしない
- 懐古趣味に陥らない（「昔はよかった」は書かない）
- 「〜ですね」「〜しましょう」など過度にカジュアルなブログ調にしない
- 「いかがでしたか？」で締めない
```

### テクニック2: 文体サンプルを3つ以上示す

Claudeの文体再現精度は、サンプル数に比例して向上する。異なるシチュエーション（回想、技術解説、読者への問いかけ）のサンプルをそれぞれ示すと、場面に応じた文体の使い分けが正確になる。

### テクニック3: 「人格設定」ではなく「経験の定義」を使う

```
効果が低い：
「あなたは52歳のベテランエンジニアです。反骨心がある性格です。」

効果が高い：
「佐藤のキャリアの年表を以下に示す。
 1998年：Slackware 3.5をインストール。cp httpd.conf httpd.conf.bakの世界。
 2003年：CVSでOSSプロジェクトに参加。SourceForge時代。
 ...
 この経験に基づいて、バージョン管理の変遷を『体験者の視点』で語ること。」
```

経験のデータを渡せば、Claudeはそこから適切な人物像を構築する。抽象的な性格描写よりも、具体的な経験の積層の方が圧倒的に効果的。

### テクニック4: Projects機能を活用する（claude.ai）

claude.aiのProjects機能を使えば、指示書をプロジェクトナレッジとして常時参照させることができる。

```
【設定手順】
1. claude.aiで新しいProjectを作成
2. Project Knowledge に指示書全文をアップロード
3. プロジェクト内の各会話で個別の回を執筆
```

これにより、毎回指示書を貼り付ける必要がなくなり、全24回を一貫したコンテキストで執筆できる。

### テクニック5: Artifactsを活用する

Claudeのファイル生成機能を使えば、Markdown/HTML/React等の形式でプレビュー可能なアーティファクトを生成できる。記事のプレビュー確認やコードの動作確認に活用できる。

## 注意点

### 事実の「もっともらしい嘘」に注意

Claudeは自信を持って不正確な事実を述べることがある。特に以下のパターンに注意。

- **年号の微妙なずれ**: 「1972年」が「1973年」になるなど
- **人名の混同**: 同じプロジェクトの複数の関係者を混同する
- **統計データの古さ**: 学習データ時点の数字を最新として述べる

Deep Researchを事前に実行し、ファクトチェック結果を会話内に保持した状態で執筆させることで、このリスクを大幅に軽減できる。

### 文体のドリフトに注意

長い会話（特に複数回の記事を連続執筆する場合）では、徐々に文体が「Claude標準」に回帰する傾向がある。3〜4回ごとに新しい会話を開始し、指示書を再投入することを推奨。

## 実際のプロンプト例（全体フロー）

```
--- 会話開始 ---

[メッセージ1: 指示書全文を投入]
（version-control.md の全文をここに貼り付け）

[メッセージ2: アウトライン依頼]
「第1回のアウトラインを作成してください」

[メッセージ3: 承認]
「良さそうです」

[メッセージ4: ファクトチェック依頼]
→ Claudeが自動的にDeep Researchを実行

[メッセージ5: 執筆依頼]
「一記事目を書いてください」

[メッセージ6: 出力依頼]
「Markdownファイルとして出力してください」

--- 会話終了（1回分完了） ---
```

| ------------------------------ | ---------------------------------- |

| ファクトチェック項目数         | 10項目（すべて一次ソース確認済み） |
| 文体の指示書準拠度             | 高（禁止事項の違反なし）           |
