# Gemini向け執筆ガイド

**Google Gemini（2.5 Pro / 2.5 Flash / Deep Research）を使った技術史連載の執筆方法**

---

## Geminiの特性と強み

Geminiは、Googleのエコシステムとの統合が最大の強みである。特に以下の特性が技術史連載の執筆に関連する。

- **超大容量コンテキストウィンドウ**: Gemini 2.5 Proは100万トークン超のコンテキストを処理可能。指示書＋過去記事＋ファクトチェック結果を一括投入できる
- **Deep Research**: Geminiの調査機能は、多段階のWeb調査を自律的に実行し、包括的なレポートを生成する
- **Google Workspace統合**: Google Docs、Drive、Gmailとの連携が可能
- **Gems（カスタムAI）**: 繰り返し使う指示をGemsとして保存できる
- **Google NotebookLM連携**: 参考文献をNotebookLMに投入し、Geminiで横断的に参照可能

---

## 推奨ワークフロー

### Step 0: Gemを作成する

GeminiのGems機能で、執筆専用のカスタムAIを作成する。これが最も効率的な起点となる。

```
【Gem設定例】

名前: 佐藤裕介の執筆エージェント

指示内容:
あなたは佐藤裕介（Engineers Hub株式会社 CEO / Technical Lead）の
執筆パートナーです。技術史連載「git ありきの世界に警鐘を鳴らす」
（全24回）の執筆を支援します。

以下のルールを厳守してください：
・一人称は「私」。「である」調で書く
・佐藤の24年間のキャリア（1990年代後半〜現在）の経験を
  「語り」として自然に挿入する
・技術的に正確であること。年号・人名・バージョンを検証可能にする
・歴史に敬意を払い、過去の技術を「劣った」と扱わない
・読者に問いかける。「あなたはどうだろうか」と投げかける
・箇条書きの羅列で終わらせず、散文で語る
・「〜と言われています」は使わない。根拠を示す
・「いかがでしたか？」で締めない

（以下、指示書の文体サンプルをここに貼り付ける）
```

### Step 1: 大容量コンテキストを活かす

Gemini最大の強みは100万トークン超のコンテキスト。これを活かし、以下をすべて一度に投入する。

```
【投入する資料】

1. 指示書全文（blueprints/version-control.md）
2. 過去に執筆済みの記事（第1回〜第N回）
   → 文体の一貫性をモデルが参照できる
3. ファクトチェック結果（research/version-control/factcheck-XX.md）
4. 参考文献の抜粋（必要に応じて）
```

```
プロンプト例：
「以下の資料を参照してください。
 [1] 執筆指示書（全文）
 [2] 第1回〜第3回の記事（文体の参考として）
 [3] 第4回のファクトチェック結果

 これらに基づいて、第4回のアウトラインを作成してください。」
```

ClaudeやChatGPTでは過去記事を全部入れるとコンテキストが圧迫されるが、Geminiなら過去記事10回分以上を同時に参照させても問題ない。

### Step 2: Deep Researchでファクトチェックを実行する

GeminiのDeep Research機能は、多段階の自律的Web調査を行い、包括的なレポートを返す。

```
プロンプト例：
「以下のトピックについて、Deep Researchで詳細に調査してください。
 各項目について、一次ソース（公式ドキュメント、論文、公式ブログ）の
 URLを必ず含めてください。

 テーマ：CVSの歴史と技術的特徴

 調査項目：
 1. Dick Gruneによる最初のCVS公開の正確な日付と公開方法
 2. Brian BerlinerによるCの書き直しの経緯と時期
 3. CVSのCopy-Modify-Mergeモデルの技術的特徴
 4. SourceForgeの設立年と初期のプロジェクト数
 5. CVSの致命的弱点（アトミックコミット不在等）の技術的根拠」
```

**ポイント**: Deep Researchは調査の「深さ」が自動調整される。項目を具体的に列挙するほど、各項目への掘り下げが深くなる。

### Step 3: Google Docsで共同編集する

Geminiで生成した記事をGoogle Docsに出力し、人間が直接編集する方法が有効。

```
プロンプト例：
「この記事をGoogle Docsに出力してください」
```

Google Docs上では、Geminiのサイドパネルを使って部分的な修正指示を出せる。

```
Docs上での修正指示例：
「選択したセクション3を、もう少し技術的に深掘りして書き直してください。
 特にCVSのリポジトリ内部構造について、ディレクトリ構成を図示してください。」
```

### Step 4: NotebookLMで参考文献を横断検索する

Google NotebookLMに参考文献（PDF、Web記事）を投入しておくと、Geminiとの連携で以下が可能になる。

```
【NotebookLMの活用方法】

1. 参考文献をNotebookLMにアップロード
   - Pro Git（PDF）
   - Version Control with Subversion（PDF）
   - Linus Torvaldsのメーリングリスト投稿（Web）
   - BitKeeper事件のタイムライン（Web）

2. NotebookLMで横断的な質問を行う
   「CVSからGitへの移行が加速した技術的・社会的要因を、
    複数の文献から横断的にまとめてください」

3. 得られた知見をGeminiの執筆に反映
```

## Gemini固有のテクニック

### テクニック1: 「思考の連鎖」を活用する（Gemini 2.5 Pro）

Gemini 2.5 Proは内部で思考の連鎖（Chain of Thought）を展開する。複雑な構成設計や、歴史的事実の整合性チェックに適している。

```
プロンプト例：
「以下の全24回の構成を分析してください。
 ステップバイステップで、以下の観点から検証してください。
 1. 各回の『問い』が論理的に積み上がっているか
 2. 歴史的な時系列が正しいか
 3. 読者の理解度が段階的に深まる構造になっているか
 4. 同じ内容が異なる回で重複していないか」
```

### テクニック2: マルチモーダル入力を活用する

Geminiは画像・PDFの直接入力に対応している。古い技術書のスキャン画像や、スクリーンショットを直接参照させることが可能。

```
プロンプト例：
（SCCSのマニュアルページのスクリーンショットを添付して）
「この画像はSCCSのmanページです。ここに記載されているコマンド体系を
 整理し、RCSとの違いをまとめてください。」
```

### テクニック3: 構造化出力を指定する

Geminiは JSON や Markdown テーブルなどの構造化出力を安定して生成できる。ファクトチェック結果を構造化データとして受け取ると、後続の執筆で参照しやすい。

```
プロンプト例：
「調査結果を以下のMarkdownテーブル形式で出力してください：

| 項目 | 事実 | ソース | URL | 信頼度 | 注意事項 |
|------|------|--------|-----|--------|---------|
| ... | ... | ... | ... | 高/中/低 | ... |」
```

## 注意点

### 日本語の文体制御が甘い場合がある

Geminiは英語での文体制御に比べて、日本語での文体制御の精度がやや低い場合がある。特に以下のパターンに注意。

- **「ですます調」への回帰**: 長文を書いていると途中で「である調」から「ですます調」に切り替わることがある
- **冗長な前置き**: 「それでは〜について見ていきましょう」のようなカジュアルな書き出しになりやすい
- **箇条書きの多用**: 散文で書くべきところを箇条書きにしてしまう傾向がある

**対策**: 禁止事項を毎回のプロンプトの冒頭に再掲する。特に「である調を維持すること」「箇条書きではなく散文で語ること」は明示的に強調する。

### Deep Researchの結果は必ず人間が検証する

Geminiの Deep Research は多数のソースを参照するが、ソース間の情報が矛盾する場合に、古い情報や不正確な二次ソースを優先することがある。特に年号と人名は必ず人間がクロスチェックする。

### コンテキストの「広さ」と「深さ」のトレードオフ

100万トークンのコンテキストに大量の資料を投入できるのはGeminiの強みだが、投入量が多すぎると個々の資料への注意が薄まる傾向がある。

**対策**: 「最も重要な参照資料」をプロンプトの冒頭で明示する。

```
プロンプト例：
「添付した資料のうち、最も優先的に参照すべきは [1] 執筆指示書です。
 文体・構成はすべてこの指示書に従ってください。
 [2] 以降はファクトチェック用の参考資料です。」
```

## 実際のプロンプト例（全体フロー）

```
--- 事前準備 ---
Gemを作成（指示書の核心部分を設定）
NotebookLMに参考文献をアップロード

--- 会話1: ファクトチェック（Deep Research） ---
[メッセージ1] 「以下の項目をDeep Researchで調査してください（項目リスト）」
→ Deep Researchが自律的に調査を実行（数分〜十数分）
[メッセージ2] 「結果をMarkdownテーブルで構造化してください」

--- 会話2: 執筆 ---
[メッセージ1] 指示書全文＋過去記事＋ファクトチェック結果を一括投入
[メッセージ2] 「第N回のアウトラインを作成してください」
[メッセージ3] 「承認します。本文を全文書いてください。10,000字以上で。」
[メッセージ4] 「Google Docsに出力してください」
→ Docs上で人間が最終編集

--- 完了 ---
```

## Claude / ChatGPT との使い分け

| 工程 | 推奨モデル | 理由 |
|------|-----------|------|
| ファクトチェック | **Gemini Deep Research** | 多段階自律調査の深さと幅 |
| 構成設計・論理検証 | Gemini 2.5 Pro / ChatGPT o3 | 推論能力の高さ |
| 本文執筆（長文） | **Claude Opus** | 文体再現精度と長文の一貫性 |
| 本文執筆（分割） | ChatGPT GPT-4o | Canvas連携の利便性 |
| 参考文献の横断分析 | **Gemini + NotebookLM** | 大容量コンテキスト＋文献管理 |
| 文体の一貫性チェック | Claude Opus | 指示書への忠実度 |

各モデルの強みを工程ごとに使い分けることで、単一モデルでは実現できない品質と効率を達成できる。
