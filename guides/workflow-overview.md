# 執筆ワークフロー全体像

**AI支援による技術史連載の執筆プロセス**

---

## はじめに

このドキュメントは、本リポジトリの連載記事がどのようなプロセスで書かれているかを解説する。AI（大規模言語モデル）を執筆パートナーとして活用しているが、AIに「書かせている」のではない。AIを「分身」として使い、人間の知識・経験・判断を拡張するためのワークフローである。

重要な前提として、**指示書の品質が出力の品質を決める**。AIに漠然と「技術記事を書いて」と指示しても、汎用的で表層的な文章しか生まれない。著者の人物像、文体、技術的バックグラウンド、連載の設計思想まで精密に定義することで、はじめて「この人が書いたとしか思えない」文章が生まれる。

---

## 全体フロー

```
Phase 1: 設計          Phase 2: 調査          Phase 3: 執筆
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ 指示書作成    │ ──→  │ ファクトチェック│ ──→  │ 本文執筆      │
│              │      │              │      │              │
│ ・著者定義    │      │ ・歴史的事実  │      │ ・アウトライン │

└──────────────┘      └──────────────┘      └──────────────┘
     人間主導              AI + 人間              AI + 人間
```

## Phase 1: 設計（指示書の作成）

### これが最も重要な工程である

指示書（blueprint）は、AIに対する「著者の分身化マニュアル」だ。以下の要素を含む。

### 1-1. 著者プロフィール

単なる経歴ではない。**著者の技術キャリアと連載テーマの関係性**を明示する。

```
悪い例：
「佐藤裕介、エンジニア歴24年、AWS・GCP・Azureに精通」

良い例：
「佐藤のキャリアはバージョン管理の歴史そのものと並走している。
 1990年代後半の手動管理 → 2000年代のCVS/SVN → 2005年のGit誕生
 → 2010年代のGit移行 → 2020年代のGitOps/AI支援開発。
 この並走経験が、連載の説得力の根幹にある。」
```

### 1-2. 文体設計

具体的な文体サンプルを複数含める。「である調」「ですます調」といった表面的な指定ではなく、思考の構造、比喩の使い方、読者への問いかけ方まで示す。

**やること・やらないことリスト**を明示する。これが文体の境界線を定義する。

### 1-3. 全体構成

全回の構成をアウトライン段階で設計する。各回のテーマ、問い、著者の体験、歴史的背景、技術論、ハンズオンを定義する。この段階で「連載全体のストーリーアーク」を確定させる。

### 1-4. 構成テンプレート

1回あたりの構成を標準化する。本リポジトリでは以下の5部構成を採用。

```
1. 導入 -- 問いの提示（1,000〜2,000字）
2. 歴史的背景（3,000〜6,000字）
3. 技術論（3,000〜6,000字）
4. ハンズオン（2,000〜4,000字）
5. まとめと次回予告（500〜1,500字）
```

## Phase 2: 調査（ファクトチェック）

### 執筆前に調査を完了させる

技術史の記事では、年号、人名、バージョン番号、組織名の正確性が信頼の根幹をなす。AIは「もっともらしい嘘」を書くことがある。執筆前にファクトチェックを独立した工程として実施し、検証済みデータだけを執筆に使う。

### 調査の対象

各回の執筆前に、以下を調査・検証する。

- **歴史的事実**: 年号、人名、組織名、ソフトウェアのバージョン
- **統計データ**: 最新の使用率、ユーザー数、市場データ
- **一次ソース**: 論文、公式ブログ、メーリングリストアーカイブ
- **技術的正確性**: コマンド・コードの動作確認

### 調査結果の記録

調査結果は `research/` ディレクトリにMarkdownで記録する。これにより、後から事実の検証・更新が可能になる。

```markdown
## 調査項目：GitHubの開発者数

- **調査日**: 2026-02-15
- **結論**: 登録開発者数1億8,000万人超（2025年10月時点）
- **一次ソース**: GitHub Octoverse 2025レポート
- **注意事項**: この数字は「登録総数」であり「月間アクティブ数」ではない
- **記事での表現**: 「GitHubの登録開発者数は2025年時点で1億8,000万人超（Octoverse 2025）」
```

## Phase 3: 執筆

### 3-1. アウトライン確認

各回の構成を事前にAIに生成させ、人間がレビューする。この段階で以下を確認する。

- 指示書で定義した「問い」が冒頭に設定されているか
- 歴史的背景のカバー範囲は適切か
- ハンズオンの難易度・実行可能性は適切か
- 前後の回との接続は自然か

### 3-2. 初稿生成

アウトラインが承認されたら、AIに本文を生成させる。指示書＋ファクトチェック結果＋アウトラインの三点セットを入力として渡す。

### 3-3. 人間によるレビュー

AIの初稿を以下の観点でレビューする。

- **事実の正確性**: ファクトチェック結果と照合
- **文体の一貫性**: 指示書で定義した文体に合致しているか
- **技術的正確性**: コマンド・コードは動作するか
- **読者体験**: 読んでいて引っかかる箇所はないか
- **著者らしさ**: 「著者が書いた」と感じられるか

### 3-4. 推敲・修正

レビュー結果に基づきAIに修正を依頼するか、人間が直接修正する。特に以下は人間が判断すべきポイント。

- 著者の個人的体験の描写の真実性
- 技術に対する評価のニュアンス
- 読者への問いかけの適切さ

各AIモデルには特性があり、同じ指示書でも指示の渡し方を最適化する必要がある。詳細は以下の個別ガイドを参照。

- [Claude向けガイド](./ai-guide-claude.md) -- 長文指示書との相性が良い。構造化された指示に忠実。
- [ChatGPT向けガイド](./ai-guide-chatgpt.md) -- 対話的に構築。Custom Instructionsの活用。
- [Gemini向けガイド](./ai-guide-gemini.md) -- 大容量コンテキスト活用。Google連携の強み。

## ファイルの命名規則

### 記事ファイル

```
{連番2桁}-{英語スラッグ}.md

例：
01-why-this-series.md
02-before-vcs.md
03-sccs-rcs.md
```

### ファクトチェック

```
factcheck-{連番2桁}.md

例：
factcheck-01.md
```

### 指示書

```
{シリーズ英語スラッグ}.md

例：
version-control.md
database-history.md
```

## 品質基準

- 各回10,000〜20,000字（日本語）
- ハンズオンのコマンドは動作確認済みであること
- 歴史的事実は一次ソースで検証済みであること
- 参考文献リストを各回末尾に付記すること
- 文体は全回を通じて一貫していること
