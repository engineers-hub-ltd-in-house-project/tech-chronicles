## 1. クラウドの請求書が教えてくれたこと

ある月末、私はクラウドの請求書を見て、しばらく動けなかった。

2019年のことだ。私が支援していたプロジェクトのAWS月額利用料が、ついに物理サーバの減価償却費と保守費用の合算を超えた瞬間だった。正確に言えば、前月比で12%増加した請求書を見て、「この増加率が続けば」と試算したとき、向こう3年のクラウド費用が、同等のオンプレミス環境を構築・運用する費用を上回ることが明らかになった。

皮肉な話だった。3年前にオンプレミスからクラウドに移行したとき、私たちは「コスト最適化」を主な理由の一つに挙げていた。物理サーバの調達に2〜3ヶ月かかっていた時代から、APIの一行でインスタンスが立ち上がる世界へ。初期費用を運用費用に転換する。需要に応じてスケールする。すべてが合理的に見えた。実際、移行後の1年目はコストが下がった。2年目は横ばいだった。そして3年目、ワークロードが安定し、事業が成長軌道に乗ったまさにそのとき、クラウド費用は加速度的に増加し始めた。

「変動費モデル」のクラウドは、ワークロードが変動しているうちは確かに経済的だ。だが、ワークロードが安定し、予測可能になった瞬間——つまり事業が成熟した瞬間——そのモデルは「毎月確実にかかる固定費を、割増料金で支払い続ける」構造に変わる。

この構造的転換に気づいたとき、私の頭に浮かんだのは、2021年にAndreessen Horowitzが発表した論文「The Cost of Cloud, a Trillion Dollar Paradox」だった。Martin CasadoとSarah Wangが上場ソフトウェア企業50社の財務報告を分析し、「大規模運用においてクラウドのコストはインフラ費用を2倍にしうる」と結論づけたあの論文だ。推定1,000億ドルの時価総額がクラウドコストの影響で失われているという試算は、当時「VCのポジショントークだ」と批判も浴びた。a16zがインフラ関連スタートアップに投資するVCであることを考えれば、批判にも一理ある。だが、私自身の請求書がその論文の正しさを証明していた。

そして2022年10月、あの宣言が出た。

Ruby on Railsの生みの親であり、37signals（Basecamp、HEYの開発元）のCTOであるDavid Heinemeier Hansson——通称DHH——が、「Why We're Leaving the Cloud」と題したブログ記事を公開したのだ。私はその記事を読んだとき、「ああ、やはりそうか」と膝を打った。DHHが書いていたことは、私が請求書を見ながら感じていたこととまったく同じだった。

「全部クラウドにすればいい」という時代は、本当に終わりつつあるのだろうか。オンプレミスに戻る企業は、何を考え、何を得て、何を失っているのか。そしてその判断は、あなたの現場にも当てはまるのだろうか。

## 2. オンプレミス回帰の系譜——誰が、なぜ、クラウドを離れたのか

### DHHの宣戦布告——37signalsのクラウド離脱

2022年10月19日に公開されたDHHのブログ記事は、技術コミュニティに衝撃を与えた。正確に言えば、衝撃を与えたのはその内容よりも、その人物が発言したという事実だった。DHHはRuby on Railsのクリエイターであり、Webアプリケーション開発の世界で最も影響力のある人物の一人だ。そのDHHが「クラウドは私たちのような企業には向いていない」と宣言したのだ。

数字は明快だった。37signalsのクラウド費用は年間320万ドルに達していた。従業員約80名、ワークロードは極めて安定的で予測可能——Basecampというプロジェクト管理ツールとHEYというメールサービスが主要プロダクトだ。トラフィックに季節変動はあるが、EC2の利用パターンは月単位でほぼ一定だった。

DHHの論点は単純明快だ。「予測可能なワークロードに対して、クラウドの『弾力性プレミアム』を払い続ける経済的合理性はない」。

2022年12月、37signalsは約60万ドル相当のDellサーバを発注した。最終的なハードウェア投資は約70万ドル。計画は野心的だった。2023年夏までに、Basecamp、HEY含む7つの主要アプリケーションをAWSおよびGoogle Cloudから自社インフラに移行する。

結果はどうだったか。

37signalsは追加の人員を雇うことなく、既存のインフラチームでこの移行を完遂した。インフラコストは半分から3分の2削減され、年間約200万ドルの節約を達成した。70万ドルのハードウェア投資は6ヶ月で回収された。5年間の節約額は1,000万ドルを超える見通しとなった。残るクラウド支出は年間約130万ドル——AWS S3の4年契約分だけだ。

この結果だけを見れば、「クラウドは高い。オンプレミスに戻るべきだ」という結論になる。だが、ここで立ち止まって考える必要がある。37signalsは極めて特殊な企業だ。従業員80名の小規模組織で、ワークロードは安定的で予測可能。グローバル展開の必要性は限定的。そして何より、DHHという傑出したエンジニアリングリーダーが自ら旗を振った。この条件のすべてが揃う企業がどれだけあるだろうか。

### Dropboxの先駆——Magic Pocket移行

37signalsより7年前、もっと大規模なクラウド離脱が実行されていた。Dropboxの「Magic Pocket」プロジェクトだ。

Dropboxは創業以来、ストレージバックエンドとしてAmazon S3を使用していた。ユーザー数の爆発的成長とともに、S3の利用料は天文学的な数字に膨れ上がっていた。Dropboxが扱うデータ量はエクサバイト規模——S3の料金体系でこの規模のストレージを運用し続けることは、事業の存続そのものに関わる問題だった。

Dropboxのエンジニアリングチームは、自社専用のストレージシステム「Magic Pocket」を開発した。ディスクの物理層からソフトウェアスタックまで、すべてを自社設計で構築するという壮大なプロジェクトだ。2015年10月7日までに、約90%のファイルがAWS S3から4つの自社データセンターに移行された。

財務的なインパクトは圧倒的だった。移行期間（2015〜2016年）にサードパーティデータセンター費用が9,250万ドル減少し、自社データセンター費用は5,300万ドル増加——差し引き3,950万ドルの純節約。2018年のIPO目論見書（S-1）では、2年間で7,460万ドルの運用コスト削減が報告された。粗利率はQ1 2016の46%からQ4 2017の70%へ、24ポイント改善した。

ただし、Dropboxのケースには重要な前提条件がある。エクサバイト規模のストレージという極めて大きく、かつ予測可能なワークロード。独自のストレージシステムを設計・実装・運用できる世界トップレベルのエンジニアリング力。そしてIPOを控えた粗利改善への強い動機。これらの条件が揃って初めて成立するプロジェクトだった。

### 2021年——a16zの「兆ドルのパラドックス」

個別企業のクラウド離脱が散発的に起きていた状況を、マクロ経済の観点から体系化したのが、2021年のAndreessen Horowitzの論文だった。

Martin CasadoとSarah Wangは、上場ソフトウェア企業50社の財務データを分析し、衝撃的な結論を導いた。大規模に運用する場合、パブリッククラウドのコストはオンプレミスの2〜3倍になりうる。そして、クラウドコストが利益率を圧迫することで、これら50社だけで推定1,000億ドルの時価総額が失われている、と。

この論文は激しい議論を巻き起こした。「VCがインフラスタートアップへの投資を正当化するためのポジショントークだ」という批判は、完全に的外れとは言えなかった。a16zはクラウド代替ソリューションに投資するVCだ。だがSADA社のRich Hoyerが指摘したように、問題の本質は「クラウドが高いかどうか」ではなく、「クラウドのコストに見合うリターンを得ているかどうか」にある。クラウドの価値はコスト削減ではなく、イノベーション速度、俊敏性、成長にある。だがその「柔軟性税」は、事業が成熟した後も払い続ける必要があるのか——これが、a16zの論文が本当に問うたことだった。

### 2024〜2025年——リパトリエーションの潮流

個別事例と学術的議論が蓄積された結果、2024年以降、クラウドリパトリエーション（クラウドからオンプレミスへの回帰）は明確なトレンドとして認識されるようになった。

Barclays CIO Survey（2024年末）によると、86%のCIOがパブリッククラウドのワークロードの一部をプライベートクラウドまたはオンプレミスに戻す計画を持っている。2020年後半の43%から倍増した数字だ。

だが、この数字を額面通りに受け取ってはならない。IDCの調査（2024年10月）によれば、全ワークロードをクラウドから移行する組織はわずか約8%にすぎない。大多数は本番データ、バックアップ、コンピュートリソースなど、特定の要素のみを移行している。「86%がリパトリエーションを計画」という見出しと、「全面撤退はわずか8%」という実態の間には、大きな隔たりがある。

そして同時に、クラウド市場全体は成長を続けている。Gartnerの予測（2024年11月発表）によると、2025年の世界パブリッククラウド支出は7,234億ドル——2024年の5,957億ドルから21.5%増だ。すべてのセグメントで二桁成長。2029年には1.42兆ドルに達する見通しである。

リパトリエーションの増加とクラウド市場の成長は、矛盾しているように見えるかもしれない。だが矛盾していない。起きているのは、「全部クラウド」という思考停止からの脱却だ。ワークロードの特性に応じて、最適な配置を選ぶ——この当たり前の判断を、企業がようやく実行に移し始めたということだ。

## 3. TCOの解剖——クラウドとオンプレミスの損益分岐点

### TCOとは何を比較しているのか

「クラウドとオンプレミス、どちらが安いか」という問いは、一見単純に見えて、実は極めて複雑だ。なぜなら、比較すべきコストの範囲が、立場によって異なるからだ。

TCO（Total Cost of Ownership：総保有コスト）は、あるインフラを「保有」するために必要な全費用を算出する手法だ。クラウドの場合は利用料がほぼすべてだが、オンプレミスの場合は表面に見えないコストが多い。

```
TCO比較: クラウド vs オンプレミスの費用構造
═══════════════════════════════════════════════════════════════

【クラウド】                    【オンプレミス】

コンピュート利用料               ハードウェア購入費（減価償却）
ストレージ利用料                 データセンター施設費
ネットワーク転送料               電力・冷却費用
マネージドサービス利用料         ネットワーク機器・回線費用
サポート契約                     OSライセンス・ソフトウェアライセンス
                                 監視・管理ツール
                                 物理セキュリティ
                                 保守・交換部品
                                 人件費（インフラエンジニア）
                                 災害対策・冗長構成

─────────────────────────────────────────────────────────────
見えやすい ← → 見えにくい
コストの可視性が異なる

═══════════════════════════════════════════════════════════════
```

クラウドのコストは請求書に集約されるため可視性が高い。一方、オンプレミスのコストは複数の部門・勘定科目に分散するため、正確な把握が難しい。この非対称性が、TCO比較を困難にする最大の要因だ。

Flexera 2025 State of the Cloud Reportは、この問題の深刻さを定量的に示している。84%の組織がクラウド支出の管理を最大の課題と回答し、クラウド予算の27%が無駄になっている。70%の組織が「クラウド予算の行き先を正確に把握できていない」と答えた。クラウドのコストは可視性が高いはずなのに、それでも70%が把握できていない——オンプレミスの真のコストを把握している組織がどれほどあるか、推して知るべしだ。

### GPU/AIワークロードが変えたTCOの方程式

クラウドとオンプレミスのTCO比較を最も劇的に変えたのは、AI/GPUワークロードの爆発的増加だ。

Lenovo Pressの「On-Premise vs Cloud: Generative AI Total Cost of Ownership（2025 Edition）」は、NVIDIA H100 GPU 8基構成のサーバについて、詳細なTCO分析を行っている。結論は明快だ。

**損益分岐点は約8,556時間——およそ11.9ヶ月。**

つまり、H100を1年以上連続稼働させるワークロードであれば、オンプレミスのほうが経済的になる。高稼働率のワークロードでは、損益分岐点は4ヶ月未満にまで短縮される。逆に、稼働率が20%未満であれば、クラウドレンタルのほうが経済的だ。

この分析が持つ意味は大きい。AI/機械学習の推論ワークロード——つまりモデルを訓練した後、本番環境で継続的に推論を実行する処理——は、まさに「予測可能で高稼働率」なワークロードの典型だ。実験段階やモデル訓練はクラウドの弾力性が活きるが、本番推論をクラウドで実行し続けることのコスト効率は、規模が大きくなるほど悪化する。

NVIDIA DGX H200/B200システム（8×GPU構成で40万〜50万ドル）の価格は決して安くない。だがFortune 100企業の半数がDGXプラットフォームでAIインフラを構築しているという事実は、大規模AI推論ワークロードにおけるオンプレミスの経済合理性を裏付けている。

### クラウドが勝つ条件、オンプレミスが勝つ条件

TCO分析から導かれる結論は、「クラウドが高い」でも「オンプレミスが安い」でもない。**条件によって答えが変わる**ということだ。

```
ワークロード特性によるインフラ選定マトリクス
═══════════════════════════════════════════════════════════════

              変動性: 高い          変動性: 低い
            ┌─────────────────┬─────────────────┐
  規模:     │                 │                 │
  大きい    │  クラウド優位   │  オンプレミス   │
            │                 │  優位           │
            │  ・季節変動大   │  ・大規模AI推論 │
            │  ・急成長期     │  ・基幹業務     │
            │  ・グローバル展開│  ・大量データ   │
            │                 │    ストレージ   │
            ├─────────────────┼─────────────────┤
  規模:     │                 │                 │
  小さい    │  クラウド優位   │  判断が分かれる │
            │                 │                 │
            │  ・スタートアップ│  ・37signals型  │
            │  ・MVP/PoC     │  ・小規模SaaS   │
            │  ・実験的       │  ・予測可能な   │
            │    ワークロード │    ワークロード │
            └─────────────────┴─────────────────┘

═══════════════════════════════════════════════════════════════
```

**クラウドが有利な条件:**

第一に、ワークロードの変動性が高い場合。ECサイトのセール期間、ゲームのイベント時、報道サイトの突発的トラフィック——需要の予測が困難で、ピークと通常時の差が大きいワークロードは、クラウドの弾力性が最大の価値を発揮する。オンプレミスでピークに合わせた設備を常時保有するコストは、クラウドの割増料金を優に上回る。

第二に、グローバル展開が必要な場合。世界中のユーザーにサービスを提供するために、複数のリージョンにインフラを配置する必要がある場合、各地にデータセンターを構えるオンプレミスのコストは現実的ではない。

第三に、急速な成長期にある場合。スタートアップや新規事業で、来月のワークロードが今月の10倍になるかもしれない——こうした状況では、オンプレミスの調達リードタイム（サーバ発注から稼働まで数週間〜数ヶ月）が致命的なボトルネックになる。

**オンプレミスが有利な条件:**

第一に、ワークロードが予測可能で安定している場合。37signalsがまさにこのケースだ。月単位でほぼ一定の負荷が続くのであれば、クラウドの弾力性プレミアムは単なる割増料金にしかならない。

第二に、大量のデータストレージが必要な場合。Dropboxのケースが示すように、ストレージはスケールするほどクラウドの単価メリットが薄れる。データの読み書きパターンが自社で最適化できるのであれば、自社設計のストレージシステムは圧倒的にコスト効率が高い。

第三に、データ規制が厳格な場合。これについては次のセクションで詳しく論じる。

### 規制という「選択の余地のない」要因

TCO比較が経済合理性の問題であるのに対し、規制対応は「選択の余地がない」要因だ。どれだけクラウドが経済的であっても、法律がオンプレミスを求めるなら、オンプレミスにするしかない。

EU GDPR（一般データ保護規則）は、EU市民のデータをEEA（欧州経済領域）内、またはそれと同等の保護水準を持つ管轄区域に保存することを要求する。違反時の制裁金は全世界売上の4%または2,000万ユーロのいずれか高い方——これは無視できる金額ではない。

さらに重要なのが、2022年12月に採択され、2025年1月17日に施行されたEU DORA（Digital Operational Resilience Act）だ。DORAは金融機関に対し、重要なICTサードパーティプロバイダーへの依存リスクを管理することを義務付けた。具体的には、重要ICTプロバイダーからの**出口戦略の策定が義務化**された。2025年11月18日、欧州監督当局はCTPP（Critical ICT Third-Party Providers）リストを公表し、Google Cloud、Microsoft、AWSが「重要プロバイダー」に指定された。

DORAが求めているのは「クラウドを使うな」ということではない。「クラウドプロバイダーが使えなくなったとき、どうやって事業を継続するか」を明文化せよ、ということだ。だが、この要求を真剣に実行しようとすれば、「すべてのインフラをAWSに依存する」という戦略は成立しなくなる。少なくとも一部のワークロードは、オンプレミスまたは代替クラウドで稼働できる状態を維持する必要がある。

米国のCLOUD法（Clarifying Lawful Overseas Use of Data Act）は、米国企業が保有するデータを、保存場所にかかわらず米国当局に提出することを義務付ける。EU GDPRとの矛盾は明らかだ。EU市民のデータをAWS（米国企業）に預けた場合、GDPRとCLOUD法の両方を同時に遵守することは、理論上不可能になりうる。

こうした規制環境は、クラウド戦略に構造的な制約を加える。データの種類と所在地を明確に管理し、規制要件に応じてオンプレミスとクラウドを使い分ける——これはもはやコスト最適化の問題ではなく、コンプライアンスの問題だ。

## 4. ハイブリッドクラウドの設計——「どちらか」ではなく「どう組み合わせるか」

### ハイブリッドクラウドの3つの設計パターン

ここまでの議論から明らかなように、「クラウドかオンプレミスか」は二者択一の問題ではない。問うべきは「どのワークロードを、どこに配置するか」だ。その答えを体系化したのが、ハイブリッドクラウドの設計パターンである。

**パターン1: クラウドバースティング**

通常はオンプレミスで稼働し、需要がピークに達したときだけクラウドにスケールアウトする。オンプレミスで「ベースロード」を処理し、クラウドで「ピーク」を吸収する設計だ。

```
クラウドバースティングの概念図
═══════════════════════════════════════════════════════════════

  負荷
  ↑
  │          ┌──┐
  │      ┌───┤  ├──┐         ← クラウドで処理
  │   ┌──┤   │  │  ├──┐        （ピーク時のみ課金）
  │───┤  │   │  │  │  ├───
  │===│==│===│==│==│==│===  ← オンプレミスの処理能力上限
  │███│██│███│██│██│██│███  ← オンプレミスで処理
  │███│██│███│██│██│██│███     （固定費）
  └───┴──┴───┴──┴──┴──┴───→ 時間

  █ = オンプレミス処理（固定費）
  ─ = クラウド処理（従量課金）

═══════════════════════════════════════════════════════════════
```

このパターンの利点は、オンプレミスの「安定したコスト」とクラウドの「弾力性」を両立できることだ。だが実装は単純ではない。オンプレミスとクラウドの間でワークロードを透過的に移動させるには、アプリケーションがステートレスであるか、状態管理が外部化されている必要がある。ネットワークのレイテンシとデータ転送コストも考慮しなければならない。

**パターン2: データティアリング**

データのアクセス頻度に応じて、保存先をオンプレミスとクラウドに分ける。頻繁にアクセスされるホットデータはオンプレミスの高速ストレージに、アクセス頻度の低いコールドデータはクラウドの低コストオブジェクトストレージに配置する。

```
データティアリングの構成例
═══════════════════════════════════════════════════════════════

  ┌─────────────────────────────────────────────┐
  │                オンプレミス                  │
  │                                             │
  │  ┌─────────────┐    ┌──────────────────┐   │
  │  │  ホットデータ │    │  ウォームデータ   │   │
  │  │  NVMe SSD    │    │  SAS HDD          │   │
  │  │  直近7日     │    │  直近90日          │   │
  │  │  高速アクセス │    │  中頻度アクセス    │   │
  │  └──────┬───────┘    └────────┬──────────┘   │
  │         │     ライフサイクル   │              │
  │         │     ポリシーで自動   │              │
  └─────────┼─────移行────────────┼──────────────┘
            │                     │
  ──────────┼─────────────────────┼──────────────
            │                     │
  ┌─────────┼─────────────────────┼──────────────┐
  │         ▼                     ▼              │
  │  ┌──────────────────────────────────────┐   │
  │  │          コールドデータ               │   │
  │  │          S3 Glacier / Azure Archive   │   │
  │  │          90日以前                      │   │
  │  │          低頻度アクセス                │   │
  │  │          低コスト（最大70%削減）       │   │
  │  └──────────────────────────────────────┘   │
  │                クラウド                      │
  └──────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
```

ライフサイクルポリシーを適切に設計すれば、長期保存コストを最大70%削減できる。金融機関や医療機関のように、規制上長期間のデータ保持が義務付けられている組織にとって、このパターンは特に有効だ。

**パターン3: ディザスタリカバリ（DR）**

主要ワークロードはオンプレミスで稼働し、クラウドをバックアップストレージおよびDRサイトとして活用する。障害時にクラウド上でワークロードを再開できる体制を維持する。

```
DR設計のティア分類
═══════════════════════════════════════════════════════════════

  Tier-1（ミッションクリティカル）:
    ├── 同期レプリケーション
    ├── RPO: ゼロ（データロスなし）
    ├── RTO: 分単位
    └── 例: 決済システム、取引システム

  Tier-2（重要業務）:
    ├── 非同期レプリケーション
    ├── RPO: 分〜時間単位
    ├── RTO: 時間単位
    └── 例: 業務アプリケーション、CRM

  Tier-3（非クリティカル）:
    ├── 定期バックアップ
    ├── RPO: 日単位
    ├── RTO: 日単位
    └── 例: 開発環境、アーカイブ

  RPO = Recovery Point Objective（どこまでのデータを保証するか）
  RTO = Recovery Time Objective（どれだけ早く復旧するか）

═══════════════════════════════════════════════════════════════
```

このパターンの要点は、すべてのワークロードを同じDRレベルで保護する必要はないということだ。Tier-1の同期レプリケーションは高コストだが、Tier-3の定期バックアップは低コストで実現できる。ワークロードの業務重要度に応じてDRティアを分類し、コストと保護レベルのバランスを取る。

### クラウドベンダーのハイブリッド戦略

クラウドベンダー自身も、ハイブリッドクラウドの需要に応えるソリューションを提供している。前回（第22回）で触れたAWS Outposts、Azure Arc、Google Anthosの2019年同時発表がその転機だった。

これらのソリューションは、クラウドの管理プレーンをオンプレミスに拡張するという発想で設計されている。AWS Outpostsはサブスクリプション型の完全マネージドサービスで、AWSがハードウェアからソフトウェア更新まで管理する。Azure Stack HCIはDell、HPE、Lenovo等のパートナーハードウェアと連携するライセンスモデルを採用し、顧客はハードウェアの選択に柔軟性を持つ。

いずれのアプローチにも、根本的な矛盾がある。クラウドベンダーが提供するオンプレミスソリューションは、結局そのベンダーへのロックインを前提としている。「オンプレミスに戻る」と言いながら、実際にはAWSの管理プレーン上でオンプレミスを運用している——これを「オンプレミス回帰」と呼べるかどうかは、議論の余地がある。

## 5. ハンズオン——TCO比較シミュレータを作る

理論だけでは、自分のワークロードにとって何が最適かを判断することはできない。ここでは、クラウドとオンプレミスのTCO比較を定量的に行うためのシミュレータを構築する。

### 演習1: TCO比較スクリプト

Bashスクリプトで、特定のワークロードシナリオにおけるクラウドとオンプレミスのTCOを比較する。

```bash
#!/bin/bash
# tco_compare.sh -- クラウド vs オンプレミス TCO比較シミュレータ
# 前提: 一般的なWebアプリケーションサーバ構成

set -euo pipefail

cat << 'HEADER'
═══════════════════════════════════════════════════════════════
  TCO比較シミュレータ: クラウド vs オンプレミス
  対象: Webアプリケーションサーバ（8 vCPU / 32GB RAM 相当）
═══════════════════════════════════════════════════════════════
HEADER

# --- パラメータ定義 ---

# クラウド（AWS EC2 m6i.2xlarge 相当、東京リージョン）
CLOUD_HOURLY_ONDEMAND=0.464       # オンデマンド料金（USD/時間）
CLOUD_HOURLY_RESERVED_1Y=0.293    # 1年リザーブド（全前払い相当）
CLOUD_HOURLY_RESERVED_3Y=0.186    # 3年リザーブド（全前払い相当）
CLOUD_STORAGE_GB_MONTH=0.096      # EBS gp3 料金（USD/GB/月）
CLOUD_STORAGE_GB=500              # ストレージ容量（GB）
CLOUD_NETWORK_GB=0.114            # データ転送料金（USD/GB、東京リージョン）
CLOUD_NETWORK_GB_MONTH=1000       # 月間データ転送量（GB）

# オンプレミス
ONPREM_SERVER_COST=8000           # サーバ購入費用（USD）
ONPREM_DEPRECIATION_YEARS=5      # 減価償却年数
ONPREM_POWER_MONTHLY=150         # 月間電力費用（USD）
ONPREM_NETWORK_MONTHLY=500       # 月間ネットワーク費用（USD）
ONPREM_DATACENTER_MONTHLY=300    # 月間データセンター費用（ラック・冷却等）
ONPREM_MAINTENANCE_YEARLY=1200   # 年間保守費用（USD）
ONPREM_ADMIN_HOURS_MONTHLY=8     # 月間管理工数（時間）
ONPREM_ADMIN_HOURLY_RATE=75      # 管理者時間単価（USD）

MONTHS=36  # 比較期間（月）

echo ""
echo "【前提条件】"
echo "  サーバ構成: 8 vCPU / 32GB RAM / 500GB SSD"
echo "  比較期間: ${MONTHS}ヶ月（$(echo "scale=1; $MONTHS / 12" | bc)年）"
echo "  月間データ転送: ${CLOUD_NETWORK_GB_MONTH}GB"
echo ""

# --- クラウドTCO計算 ---
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "【クラウド TCO（${MONTHS}ヶ月）】"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"

# オンデマンド
cloud_compute_od=$(echo "scale=2; $CLOUD_HOURLY_ONDEMAND * 730 * $MONTHS" | bc)
cloud_storage=$(echo "scale=2; $CLOUD_STORAGE_GB_MONTH * $CLOUD_STORAGE_GB * $MONTHS" | bc)
cloud_network=$(echo "scale=2; $CLOUD_NETWORK_GB * $CLOUD_NETWORK_GB_MONTH * $MONTHS" | bc)
cloud_total_od=$(echo "scale=2; $cloud_compute_od + $cloud_storage + $cloud_network" | bc)

echo "  オンデマンド:"
printf "    コンピュート:   \$%'.2f\n" "$cloud_compute_od"
printf "    ストレージ:     \$%'.2f\n" "$cloud_storage"
printf "    ネットワーク:   \$%'.2f\n" "$cloud_network"
printf "    合計:           \$%'.2f\n" "$cloud_total_od"
echo ""

# 1年リザーブド
cloud_compute_1y=$(echo "scale=2; $CLOUD_HOURLY_RESERVED_1Y * 730 * $MONTHS" | bc)
cloud_total_1y=$(echo "scale=2; $cloud_compute_1y + $cloud_storage + $cloud_network" | bc)

echo "  1年リザーブド:"
printf "    コンピュート:   \$%'.2f\n" "$cloud_compute_1y"
printf "    合計:           \$%'.2f\n" "$cloud_total_1y"
echo ""

# 3年リザーブド
cloud_compute_3y=$(echo "scale=2; $CLOUD_HOURLY_RESERVED_3Y * 730 * $MONTHS" | bc)
cloud_total_3y=$(echo "scale=2; $cloud_compute_3y + $cloud_storage + $cloud_network" | bc)

echo "  3年リザーブド:"
printf "    コンピュート:   \$%'.2f\n" "$cloud_compute_3y"
printf "    合計:           \$%'.2f\n" "$cloud_total_3y"

# --- オンプレミスTCO計算 ---
echo ""
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "【オンプレミス TCO（${MONTHS}ヶ月）】"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"

onprem_depreciation=$ONPREM_SERVER_COST
onprem_power=$(echo "scale=2; $ONPREM_POWER_MONTHLY * $MONTHS" | bc)
onprem_network=$(echo "scale=2; $ONPREM_NETWORK_MONTHLY * $MONTHS" | bc)
onprem_datacenter=$(echo "scale=2; $ONPREM_DATACENTER_MONTHLY * $MONTHS" | bc)
onprem_maintenance=$(echo "scale=2; $ONPREM_MAINTENANCE_YEARLY * $MONTHS / 12" | bc)
onprem_admin=$(echo "scale=2; $ONPREM_ADMIN_HOURS_MONTHLY * $ONPREM_ADMIN_HOURLY_RATE * $MONTHS" | bc)
onprem_total=$(echo "scale=2; $onprem_depreciation + $onprem_power + $onprem_network + $onprem_datacenter + $onprem_maintenance + $onprem_admin" | bc)

printf "    サーバ購入費:   \$%'.2f\n" "$onprem_depreciation"
printf "    電力費用:       \$%'.2f\n" "$onprem_power"
printf "    ネットワーク:   \$%'.2f\n" "$onprem_network"
printf "    DC費用:         \$%'.2f\n" "$onprem_datacenter"
printf "    保守費用:       \$%'.2f\n" "$onprem_maintenance"
printf "    管理工数:       \$%'.2f\n" "$onprem_admin"
printf "    合計:           \$%'.2f\n" "$onprem_total"

# --- 比較結果 ---
echo ""
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "【比較結果（${MONTHS}ヶ月）】"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
printf "  クラウド（オンデマンド）: \$%'.2f\n" "$cloud_total_od"
printf "  クラウド（1年RI）:       \$%'.2f\n" "$cloud_total_1y"
printf "  クラウド（3年RI）:       \$%'.2f\n" "$cloud_total_3y"
printf "  オンプレミス:            \$%'.2f\n" "$onprem_total"
echo ""

# 損益分岐点の概算
cloud_monthly_od=$(echo "scale=2; $cloud_total_od / $MONTHS" | bc)
onprem_monthly_after_purchase=$(echo "scale=2; ($onprem_total - $onprem_depreciation) / $MONTHS" | bc)
if [ "$(echo "$cloud_monthly_od > $onprem_monthly_after_purchase" | bc)" -eq 1 ]; then
  breakeven=$(echo "scale=1; $onprem_depreciation / ($cloud_monthly_od - $onprem_monthly_after_purchase)" | bc)
  echo "  損益分岐点（vs オンデマンド）: 約${breakeven}ヶ月"
fi

echo ""
echo "※ この試算は単一サーバ・単一ワークロードの概算です"
echo "  冗長構成、スケーリング要件、管理ツールのコスト等は含みません"
echo "═══════════════════════════════════════════════════════════════"
```

### 演習2: ワークロード分類ツール

自社のワークロードがクラウドとオンプレミスのどちらに適しているかを判断するための分類ツールを構築する。

```bash
#!/bin/bash
# workload_classifier.sh -- ワークロード配置判定ツール
set -euo pipefail

cat << 'HEADER'
═══════════════════════════════════════════════════════════════
  ワークロード配置判定ツール
  各ワークロードの特性を入力し、最適な配置を判定します
═══════════════════════════════════════════════════════════════
HEADER

# スコアリング関数
classify_workload() {
  local name="$1"
  local variability="$2"     # 負荷変動性: 1(安定)-5(激変)
  local predictability="$3"  # 予測可能性: 1(不可能)-5(完全に予測可能)
  local data_sensitivity="$4" # データ機密性: 1(公開可)-5(極秘)
  local scale="$5"           # 規模: 1(小)-5(大)
  local growth_rate="$6"     # 成長率: 1(横ばい)-5(急成長)

  # クラウドスコア: 変動性が高い、成長率が高い → クラウド向き
  local cloud_score=$(( variability * 2 + growth_rate * 2 + (6 - predictability) ))

  # オンプレスコア: 予測可能、データ機密性が高い、規模が大きい → オンプレ向き
  local onprem_score=$(( predictability * 2 + data_sensitivity * 2 + scale ))

  echo "  ワークロード: ${name}"
  echo "    変動性=${variability} 予測可能性=${predictability} \
機密性=${data_sensitivity} 規模=${scale} 成長率=${growth_rate}"
  echo "    クラウドスコア: ${cloud_score} / オンプレスコア: ${onprem_score}"

  if [ "$cloud_score" -gt "$((onprem_score + 3))" ]; then
    echo "    → 推奨: クラウド"
  elif [ "$onprem_score" -gt "$((cloud_score + 3))" ]; then
    echo "    → 推奨: オンプレミス"
  else
    echo "    → 推奨: ハイブリッド（要詳細分析）"
  fi
  echo ""
}

echo ""
echo "【サンプルワークロードの分析】"
echo ""

# サンプルワークロード
classify_workload "ECサイト（セール期間あり）" 5 2 3 3 4
classify_workload "社内基幹業務システム"       1 5 4 4 1
classify_workload "AI推論サービス（本番）"     2 4 3 5 2
```

### 演習3: 損益分岐点の可視化

月ごとの累計コストを比較し、損益分岐点を特定する。

```bash
#!/bin/bash
# breakeven_chart.sh -- 損益分岐点の可視化
set -euo pipefail

cat << 'HEADER'
═══════════════════════════════════════════════════════════════
  損益分岐点チャート: クラウド vs オンプレミス
  36ヶ月間の累計コスト推移
═══════════════════════════════════════════════════════════════
HEADER

# パラメータ（8vCPU/32GB構成）
CLOUD_MONTHLY=339       # クラウド月額（オンデマンド、USD）
ONPREM_INITIAL=8000     # オンプレ初期投資（USD）
ONPREM_MONTHLY=520      # オンプレ月額運用費（USD）

echo ""
echo "  前提: クラウド月額 \$${CLOUD_MONTHLY} / オンプレ初期 \$${ONPREM_INITIAL} + 月額 \$${ONPREM_MONTHLY}"
echo ""
echo "  月  | クラウド累計 | オンプレ累計 | 差額        | グラフ"
echo "  ----+-------------+-------------+-------------+------------------"

breakeven_found=0
for month in $(seq 1 36); do
  cloud_total=$((CLOUD_MONTHLY * month))
  onprem_total=$((ONPREM_INITIAL + ONPREM_MONTHLY * month))
  diff=$((cloud_total - onprem_total))

  # 簡易棒グラフ
  cloud_bar_len=$((cloud_total / 500))
  onprem_bar_len=$((onprem_total / 500))

  cloud_bar=""
  for i in $(seq 1 "$cloud_bar_len"); do cloud_bar="${cloud_bar}#"; done
  onprem_bar=""
  for i in $(seq 1 "$onprem_bar_len"); do onprem_bar="${onprem_bar}="; done

  marker=""
  if [ "$diff" -ge 0 ] && [ "$breakeven_found" -eq 0 ]; then
    # まだ損益分岐前
    true
  fi
  if [ "$diff" -lt 0 ] && [ "$breakeven_found" -eq 0 ]; then
    breakeven_found=1
  fi

  if [ $((month % 3)) -eq 0 ] || [ "$month" -eq 1 ]; then
    printf "  %2d  | \$%'9d | \$%'9d | \$%'9d |" \
      "$month" "$cloud_total" "$onprem_total" "$diff"
    if [ "$diff" -lt 0 ]; then
      echo " オンプレ有利"
    else
      echo " クラウド有利"
    fi
  fi
done

echo ""
echo "  # = クラウド累計コスト"
echo "  = = オンプレミス累計コスト"
echo ""

# 損益分岐点の計算
if [ "$((CLOUD_MONTHLY - ONPREM_MONTHLY))" -gt 0 ]; then
  # この場合クラウドの方が月額が安いので、オンプレの初期投資を回収できない
  echo "  結果: この構成ではクラウド月額がオンプレ月額より低いため、"
  echo "        オンプレミスの初期投資は回収できません"
else
  breakeven_month=$(echo "scale=1; $ONPREM_INITIAL / ($CLOUD_MONTHLY - $ONPREM_MONTHLY)" | bc 2>/dev/null || echo "N/A")
  echo "  損益分岐点: 約${breakeven_month}ヶ月"
  echo "  → ${breakeven_month}ヶ月以上運用するなら、オンプレミスが経済的"
fi

echo ""
echo "═══════════════════════════════════════════════════════════════"
echo "注: 実際のコストはリザーブドインスタンス、Savings Plans、"
echo "    冗長構成、スケーリング要件等により大きく変動します"
echo "═══════════════════════════════════════════════════════════════"
```

これらのスクリプトを実行してみてほしい。自社のワークロードのパラメータを入力し、クラウドとオンプレミスのどちらが経済的かを定量的に分析する。数字は嘘をつかない。だが、数字だけでは判断できないこと——組織のスキルセット、成長戦略、規制環境——があることも忘れないでほしい。

## 6. まとめと次回予告

### 「全部クラウド」の終わり、「選んで使う」の始まり

第23回では、オンプレミス回帰の潮流を、事例・コスト構造・規制・設計パターンの4つの観点から解剖した。

改めて整理しよう。

第一に、オンプレミス回帰は実在するが、全面撤退ではない。37signalsやDropboxの事例は鮮烈だが、IDCの調査が示すように全ワークロードをクラウドから移す組織はわずか8%だ。起きているのは「全部クラウド」からの脱却であり、「全部オンプレミス」への回帰ではない。

第二に、TCOの方程式はワークロード特性で決まる。変動性が高く成長中のワークロードにはクラウドが有利。予測可能で安定したワークロードにはオンプレミスが有利。GPU/AIワークロードでは、約12ヶ月の連続稼働で損益分岐点に達する。

第三に、規制は「選択の余地のない」制約だ。GDPR、DORA、各国のデータローカリゼーション法は、コスト以前の問題としてインフラ配置に制約を加える。特にDORAが求める「出口戦略」は、単一クラウドへの全面依存を構造的に困難にする。

第四に、ハイブリッドクラウドは「妥協」ではなく「設計」だ。バースティング、ティアリング、ディザスタリカバリ——それぞれのパターンは、ワークロードの特性に応じた合理的な配置戦略だ。

冒頭の問いに戻ろう。「全部クラウドにすればいい」時代は終わりつつあるのか。私の答えはこうだ。「全部クラウドにすればいい」という思考停止が終わりつつある。クラウドは依然として強力なインフラ選択肢だ。Gartnerが予測する7,234億ドルのクラウド支出がそれを物語っている。だが、クラウドは万能ではない。オンプレミスも万能ではない。ワークロードの特性を理解し、コスト構造を分析し、規制要件を把握し、組織の能力を見極めた上で、最適な配置を「選ぶ」。この判断力こそが、クラウド時代のインフラエンジニアに求められる能力だ。

あなたの現場では、どのワークロードがクラウドに適し、どのワークロードがオンプレミスに適しているだろうか。その判断の根拠を、定量的に説明できるだろうか。

### 次回予告

次回、第24回（最終回）「計算資源の民主化——『他人の計算機を借りる』とは何だったのか」では、メインフレームからサーバーレスまでの全24回を俯瞰する。計算資源の民主化を貫く3つの原則——抽象化、弾力性、共有——を軸に、私たちが70年間で何を得て何を失ったかを問い直す。そして、次の革命——量子コンピューティングのクラウド提供、AI専用インフラの普及——に対して、問いを知る者だけが適応できるという、この連載の核心メッセージに立ち返る。

24年間インフラを触り続けてきたエンジニアとして、最後に伝えたいことがある。
